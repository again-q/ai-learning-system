## 一、项目起源

重点高中普遍部署了校园学情管理系统，能动态追踪每个学生在每个知识点上的掌握情况。但这些系统的服务对象是**教师**——用于个性化出卷、学情分析、教学调整。学生本人依然处于信息黑箱之中：只知道分数，不知道「哪里不会」「为什么不会」「下一步该学什么」。

本项目的初衷，是把这种学情分析能力从教师侧搬到学生侧，让学生真正掌握自己的学习情况。

### 发起人背景

项目负责人现为初升高学生，深刻体会过课内学习中的真实痛点：题做了很多却说不清自己哪里会、哪里不会；考试失分后只能归因到「粗心」「没复习到」，无法定位到具体的知识盲区；想针对性补强却不知道从何下手。这些困惑的根源，是学生缺少一个能客观刻画自身学习情况的工具。

本项目正是基于这一亲历的痛点发起——以五维能力向量模型量化每个知识点的真实掌握程度，以 AI 对话支持针对性深入提问，与课内学习形成相辅相成的闭环。

## 二、系统目标

### 定位

**课后辅助工具，而非替代课堂学习。**

帮助学生巩固、查漏、规划，不代替学校教学。与课内学习形成相辅相成的关系——课堂学新知，课后用本系统消化吸收、查漏补缺、针对性深入。

### 核心目标

| 目标 | 说明 |
|------|------|
| **量化真实掌握** | 用五维向量（K/A/T/Q/S）追踪每个知识点的真实掌握程度，而非简单统计做题数 |
| **动态学习路径** | 根据薄弱点 AI 自动调整学什么、什么时候复习 |
| **全科统一管理** | 一个系统管理所有学科的知识图谱和掌握度 |
| **认知分层闭环** | 从学到练到测到补，形成完整学习链路，解决学习生活中的遗漏点 |

### 价值主张

一条链路打通学生学习生活：**了解学习情况 → 定位不懂之处 → 深入提问突破 → 与课内学习呼应**。

## 四、已实现：五维能力分析（知识图谱页）

作为系统的首个能力模块，知识图谱页采用**五维能力向量模型**对学科能力进行精细刻画：

| 维度 | 符号 | 含义 |
|------|------|------|
| Knowledge | K | 知识储备（概念、公式、定理） |
| Application | A | 知识应用（标准题型解题） |
| Thinking | T | 思维能力（推理、归纳、迁移） |
| Problem Solving | Q | 问题解决（综合题、开放题） |
| Speed | S | 解题速度与准确率 |

向量 `M = (K, A, T, Q, S)` 独立评估各维度，避免「一个总分掩盖薄弱点」。通过 Canvas 自绘中心辐射图 + 镜头缩放交互，让能力分布一目了然。

> 详细理论见 [五维能力向量框架-理论文档.md](五维能力向量框架-理论文档.md)

## 五、知识库架构（规划中）

知识库是五维量化能力的工程基础，采用**「模板 + 个人修改」架构**。

### 设计原则

- **知识库是模板，不是真理**：官方提供学科知识库作为起点（如数学库），学生基于自身情况增删改
- **修改仅对个人生效**：所有修改保存在个人副本中，不影响全局，零污染零审核
- **尊重个体差异**：教材版本、学校进度、个人薄弱点因人而异，强制统一反而失真

### 数据流

```
┌─────────────────────────────────┐
│     官方模板知识库（只读）        │
│  由维护者构建（如数学知识库）     │
└──────────────┬──────────────────┘
               │ 用户首次使用时 fork
               ▼
┌─────────────────────────────────┐
│       个人知识库副本             │
│  可增 / 删 / 改，仅本人可见      │
│  每个知识点标记来源：             │
│    · template  来自模板          │
│    · personal  个人新增          │
└─────────────────────────────────┘
```

### 新增知识点的五维评估规则

新增节点不能立刻有五维分数（否则等于主观自评，失去量化意义），采用**三阶段演进**：

| 阶段 | 五维状态 | 触发条件 | 处理方式 |
|------|---------|---------|---------|
| 新增时 | `null`（未评估） | 用户/AI 添加节点 | 图谱显示为虚线灰色节点，**不参与**总掌握度计算 |
| 首次激活 | 算法初始化五维 | 学生首次在该知识点上做题/对话/标记学习 | AI 根据首次交互行为给出初始估计 |
| 持续更新 | 动态修正 | 每次相关学习行为 | BKT/IRT 算法持续调整分数 |

**关键设计**：用 `null` 而非 `0` 表示未评估。`0` 意味着「学了但完全不会」，`null` 意味着「还没学」，计算总掌握度时必须区分。

### 新增知识点的轻量化依据机制

**子知识点新增是必要能力**——学生在学习中常发现「二次函数求根公式」是「二次函数」的子节点，需要能补充进图谱。但为防止乱加导致体系崩坏，新增时需提供依据与内容，**流程必须轻量**。

**必填字段（极简）**：

| 字段 | 说明 | 约束 |
|------|------|------|
| 知识点名称 | 如「二次函数求根公式」 | ≤20 字，必填 |
| 父级知识点 | 挂到哪个现有节点下 | 必选，自动建立图谱连接 |
| 依据 | 为什么认为这是一个知识点 | 一句话，≤50 字，必填 |
| 内容描述 | 这个知识点讲什么 | 一段话，≤200 字，必填 |

**轻量化体现在**：
- 全部字段支持语音输入，学生口述即可，不必逐字打字
- 「依据」与「内容描述」合并为一个输入框的智能识别——学生写一段自然语言，AI 自动拆分为依据和内容
- 父级知识点用搜索+联想选择，不用手动输入

### 防止知识体系崩坏的 AI 软审核

不设硬性拦截（避免打击共建积极性），改用 **AI 软审核 + 主动建议**：

- 新增提交时，AI 判断：是否为有效知识点、是否与父级有逻辑关联、是否与现有节点重复
- **重复时**：AI 主动建议合并（如「二次函数求根」→ 合并到「二次函数」）
- **存疑时**：标记为「待确认」，图谱显示但五维不纳入计算，提示用户复核
- **有效时**：直接入库，进入「未评估」状态，等待首次激活

**结构约束**（仅 1 条硬规则）：
- 每个新节点必须挂到一个父级节点上，禁止孤立节点——这是图谱完整性的底线

### 当前进度

数学知识库由项目负责人构建中，作为首个官方模板。其他学科先搭建框架，后续逐步填充。

## 六、快速开始

### 环境要求

- 微信开发者工具（最新稳定版）
- 微信小程序 AppID
- 已开通微信云开发

### 运行步骤

1. **克隆仓库**

   ```bash
   git clone https://github.com/again-q/ai-learning-system.git
   ```

2. **导入项目**

   微信开发者工具 → 导入项目 → 选择仓库根目录 → 填入 AppID

3. **开通云开发**

   工具栏点击「云开发」按钮开通，记下**云环境 ID**

4. **配置云环境**

   编辑 [miniprogram/app.js](miniprogram/app.js)，替换 `env`：

   ```js
   wx.cloud.init({
     env: 'your-cloud-env-id',
     traceUser: true
   });
   ```

5. **部署云函数**

   右键 `cloudfunctions/userLogin` → 「上传并部署：云端安装依赖」

6. **初始化数据库**

   云开发控制台创建 `users` 集合（权限：仅创建者可读写）

7. **预览**

   点击工具栏「预览」或「真机调试」

## 七、数据模型

`users` 集合主要字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `_openid` | string | 微信 OpenID（主键） |
| `nickName` | string | 昵称 |
| `avatarUrl` | string | 头像 fileID |
| `streak` | number | 连续学习天数 |
| `subjects` | array | 已选学科 |
| `kOverall` | number | 知识维度总分（五维预留字段） |
| `aOverall` | number | 应用维度总分（五维预留字段） |
| `tOverall` | number | 思维维度总分（五维预留字段） |
| `qOverall` | number | 问题解决总分（五维预留字段） |
| `sOverall` | number | 速度维度总分（五维预留字段） |
| `createdAt` | date | 注册时间 |
