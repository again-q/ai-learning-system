# Task Decomposer

将系统架构文档（SAD）拆解为完整详细设计文档体系。输入：`doc/arch/`；输出：`doc/detailed/`。

## 关键概念

| 产出物 | 本质 | 读者 | 强制性 |
|-------|-----|-----|-------|
| `编码规范.md` | **怎么写代码**的风格指南（命名、格式、注释） | 开发者 | 建议遵守 |
| `{模块}_编码指令.md` | **每模块的刚性编码依据**（文件清单、API 契约、BR 规则、禁止项） | gatekeeper | **防漂移核心依据** |

## 工作流

### Step 0：启动检测（每次第一步）

**检测1：恢复模式** — 检查 `doc/detailed/_PROGRESS.md` 是否存在且有 `⏳` 条目：
- **有** → 读取文件，告知用户"从 [第一个⏳文档] 继续"，直接开始，无需确认
- **无** → 执行检测2

**检测2：端类型探测** — 按优先级：
1. `doc/arch/{项目名}_SAD_概览.md` → 获取端类型
2. `doc/prd/{项目名}_PRD_概览.md` → 获取端类型
3. `doc/arch/` 目录：`*_SAD_前端.md` → Web前端；`*_SAD_小程序.md` → 小程序；仅有 `*_SAD_后端.md` 或 `*_SAD.md` 且无前端/小程序文件 → 纯后端
4. 用户对话中明确提及 → 直接使用
5. 无法确定 → 询问用户

**端类型 → 执行步骤：**

| 端类型 | 执行步骤 |
|-------|---------|
| 纯后端 | 0→1→2→2.5→3→5→6 |
| 后端+Web前端 | 0→1→2→2.5→3→4（前端）→5→6 |
| 后端+小程序 | 0→1→2→2.5→3→4（小程序）→5→6 |
| 三端 | 0→1→2→2.5→3→4（前端+小程序）→5→6 |

**失败动作**：若无法探测端类型且用户未说明，询问用户后等待回复，不得假设并继续执行。

---

### Step 1：架构分析 + 创建进度文档

解析架构文档，识别模块/API/数据模型/技术栈，向用户展示拆解方案（后端模块 + 前端页面域 + 小程序页面域）。

**⛔ 前端/小程序端接口需求正向比对（含前端或小程序时必须执行，在展示拆解方案前完成）**：

架构文档中前端/小程序 SAD 或 PRD 里明确要求的接口，后端可能未规划对应实现。此比对是发现"需求存在但后端缺失"这类断链的唯一入口，不可省略：

1. **提取需求侧接口期望清单**：
   - 扫描 `doc/arch/{项目名}_SAD_前端.md` / `doc/arch/{项目名}_SAD_小程序.md` 的「页面交互」「数据流」「接口调用」等章节，提取所有明确提及的接口（含 URL、功能描述、调用时机）
   - 若存在 `doc/prd/` 目录，同样扫描其中的接口/API 引用
   - 汇总为「需求侧期望接口清单」

2. **与后端已规划接口做正向比对**：
   - 从 `doc/arch/{项目名}_SAD_后端.md` 各模块设计章节提取后端已规划接口清单
   - 逐条核对「需求侧期望接口清单」中的每条接口，是否在后端已规划清单中有**路径和方法均匹配**的项
   - **若发现"需求侧期望但后端未规划"的接口**：在 `_PROGRESS.md` 中单独列为「⚠️ 规则零缺口：需求驱动待补充接口」条目，并在 Step 2.5 中作为额外检查项强制补充（优先级高于规则一~六，原因：这是需求明确要求的、不是推导出来的）

3. **展示比对结果**：
   - 在拆解方案中单独列出「需求驱动缺口」小节，列出所有 `⚠️ 规则零缺口` 接口及其来源文档
   - 若前端/小程序 SAD 无接口引用信息，注明「前端/小程序 SAD 未包含接口引用，正向比对无输入，仅依赖 Step 2.5 规则一~六推导」

用户确认后，**严格按以下顺序执行，不得调换**：

1. **立即**加载 `templates/backend-template.md` Part 2
2. **立即**写入 `doc/detailed/_PROGRESS.md` ← **这是整个流程的第一个文件写入操作，必须在生成任何详设文档之前完成**
3. 写入成功后，才允许进入 Step 2

> ⛔ **硬性规则**：`_PROGRESS.md` 未写入前，禁止写入任何其他文件。违反此规则视为流程错误，必须立即停止并补创建进度文档。

**失败动作**：若 `_PROGRESS.md` 写入失败，立即停止，告知用户文件路径和错误信息，等待确认后再重试，不得跳过。

### Step 2：依赖排序

加载 `resources/design-guides.md` Part 1，按 LC-001 语言为每个后端文档分配层次（Layer 0-4），建立构建顺序。第12节依赖关系必须精确到接口路径。

**失败动作**：若架构文档缺少必要的模块信息导致无法排序，告知用户缺少哪个模块的信息，等待用户补充，不得假设依赖关系。

---

### Step 2.5：功能链完整性推导

> ⛔ **本步骤不可跳过，不可简化，不可与 Step 2 合并执行。**

**执行前提：** 加载 `resources/design-guides.md` Part 2，完整读取后再开始扫描。

**强制扫描流程（严格按顺序，逐模块、逐规则执行）：**

**第一步：收集接口清单**
- ⛔ **必须**从架构文档的每个模块设计章节提取已知接口，列出：HTTP方法 + 路径 + 功能描述
- ⛔ **禁止**跳过任何一个模块，即使该模块接口数量少于3个

**第二步：逐规则扫描（规则零 → 规则六，全部必须执行）**

| 规则 | 扫描内容 | 不通过时的强制动作 |
|------|---------|-----------------|
| **规则零：需求驱动缺口（优先）** | 读取 `_PROGRESS.md` 中「⚠️ 规则零缺口：需求驱动待补充接口」列表，逐条确认对应后端模块的接口清单中已有该接口 | ⛔ 未有则立即补充到对应模块接口清单；若无法确定归属模块则新建专用模块条目 |
| **规则一：操作前置依赖** | 每个 `PUT`/`DELETE`/`PATCH` 接口是否有对应的 `GET /{resource}/{id}` | ⛔ 必须补充缺失的 GET 接口 |
| **规则二：状态机完整性** | 有状态字段的模块：① 状态枚举定义 ② 每个状态变更接口的伪代码是否含前置状态校验 ③ 是否有驳回后重新提交的逆向路径 | ⛔ 必须补充枚举定义、校验伪代码、逆向接口 |
| **规则三：跨模块依赖（声明阶段）** | 列出本模块第12节依赖的每个外部接口路径，标注「待验证」 | ⛔ 必须逐条列出依赖路径，禁止笼统写「依赖用户模块」 |
| **规则四：数据生命周期** | 有「创建关联」「草稿」「导入/导出」「软删除」场景的模块，是否有对应的逆向或查询接口 | ⛔ 必须补充解绑/恢复/草稿列表/任务状态查询接口 |
| **规则五：异步流程** | 有 MQ 消费者、定时任务、异步导入/导出的模块，是否有任务状态查询接口和死信处理方案 | ⛔ 必须补充 `GET /tasks/{id}/status` 和死信处理伪代码 |
| **规则六：权限与数据隔离（意图识别）** | 从架构文档数据模型章节推断是否有多租户/数据归属设计（`tenant_id`/`owner_id`/`created_by`） | ⛔ 若识别到多租户/归属设计，必须标注「Step 3 生成 DDL 后补充执行规则六验证」 |

> ⛔ **规则零~六共7条规则必须全部扫描，不得以「本模块无此场景」为由跳过，必须明确写出「已检查，无此场景」。规则零为最高优先级，来自明确需求要求。**

**第三步：输出推导报告并持久化**
- ⛔ **必须**逐模块输出推导报告（使用 `resources/design-guides.md` Part 2 末尾的「推导报告格式」）
- ⛔ **每条规则必须有明确结论**：`✅ 规则X：已满足` 或 `✅ 规则X：已满足（本模块无此场景）` 或 `⚠️ 发现缺失 → 已补充到接口清单`
- ⛔ **报告末尾必须统计**：共补充 N 个接口，M 处伪代码逻辑
- ⛔ **报告展示后，立即将补充接口写入 `_PROGRESS.md`**

**第四步：用户确认**
- 向用户展示推导报告，等待确认
- ⛔ **用户确认前，禁止进入 Step 3**
- 若用户明确要求跳过 Step 2.5，AI 应警告：「跳过功能链推导可能导致生成的详设文档存在断链问题，建议至少执行规则一（操作前置依赖）检查。是否确认跳过？」用户**在本轮对话中**明确回复「确认跳过」或「是」后方可跳过，其他任何回复（包括沉默、「快点」、「继续」等）均视为未确认，不得跳过。

**第五步：锁定最终接口清单**
- ⛔ **Step 3 生成文档时，必须以本步骤确认后的接口清单为准**，禁止在 Step 3 中自行增减接口
- ⛔ **锁定范围仅限第3节（对外暴露的 API 接口）**；Step 3 第7节（外部接口）和第8节（内部接口）不受锁定约束
- ⛔ **Step 3 每个文档的第1节（功能描述）和第3节（接口定义）必须包含本步骤补充的所有接口**，不得遗漏

**失败动作**：若推导过程中发现架构文档有严重歧义（如模块边界不清晰），停止推导，向用户说明具体歧义点，等待澄清后再继续。

### Step 3：后端详设生成（逐文档写入）

加载 `templates/backend-template.md` Part 1。

> ⛔ **单文档节奏硬性规则（防止上下文溢出与重复生成）**：
> **每次只生成一份文档**，写入成功并更新 `_PROGRESS.md` 后，必须暂停并向用户输出如下格式的进度确认消息，等待用户明确回复"继续"后方可生成下一份文档：
>
> ```
> ✅ [{模块}_{功能域}.md] 已写入。_PROGRESS.md 已更新（{当前进度，如 2/7}）。
> 📋 剩余：{列出所有 ⏳ 状态的文档名}
> 👉 请回复「继续」生成下一份，或「暂停」结束本次会话（下次可自动恢复）。
> ```
>
> **禁止**在用户确认前自动生成下一份文档。**禁止**在单次回复中连续写入多份文档。
> **唯一例外**：用户在启动时明确说「全部自动生成，无需逐个确认」，则视为批量授权，但每写完一份仍须更新 `_PROGRESS.md`。

**每个文档的写入顺序：先将文档内容完整写入文件，写入成功后立即更新 `_PROGRESS.md`（⏳→✅），再等待用户确认，之后才生成下一个。禁止先更新进度再写文档。**

**每个文档生成前的前置动作（必须执行）：**
1. 读取 `_PROGRESS.md` 中本模块的「补充接口（Step 2.5）」子项，将补充接口纳入本文档第1节和第3节
2. **规则三动态验证**：检查 Step 2.5 推导报告中本模块声明的「待验证」依赖接口，在**已生成的**其他模块文档中查找是否存在；若存在则标注「✅ 已验证」，若不存在则在本文档第12节标注「⚠️ 被依赖接口待补充：{接口路径}」
3. **规则六动态验证**（仅当 Step 2.5 推导报告中本模块标注了「Step 3 生成 DDL 后补充执行规则六验证」时）：生成 DDL 后，检查是否含 `tenant_id`/`owner_id`/`created_by` 字段；若有，检查每个查询和写操作的伪代码是否含隔离/校验步骤

**12个必填节：** 功能描述 / 业务规则 / 接口定义（OpenAPI 3.0 YAML）/ 功能逻辑（伪代码）/ 算法 / DDL / 外部接口 / 内部接口 / 性能要求 / 安全要求 / 测试要点 / 依赖关系

> ⛔ **第3节接口定义硬性规则（违反即重写该节）**：
> - **必须**使用 ` ```yaml ` 代码块，严格遵循 `templates/backend-template.md` 第3节的 OpenAPI 3.0 格式
> - **必须**包含：HTTP 方法、路径、requestBody（含 schema/required/properties/example）、responses（含 200/400/401/500 的 schema 和错误码示例）
> - **禁止**用 Markdown 表格替代 YAML 格式
> - **禁止**用纯文字描述替代 YAML 格式
> - **禁止**省略 responses 中的错误响应（400/401/500 必须存在）

**失败动作**：若某份文档写入失败，立即停止，告知用户失败的文档名和原因，等待用户确认后再重试或跳过，不得继续生成后续文档。

### Step 4：前端 / 小程序详设（按端类型执行）

**Web 前端**（LC-FE-001 ≠ `无` 时执行）：
加载 `templates/frontend-template.md` Part 1 + `resources/design-guides.md` Part 3。
10个必填节：页面清单 / 页面交互规则 / API调用映射 / 状态定义 / 组件拆分 / 复用逻辑 / 路由设计 / 错误处理 / 构建与环境配置 / 测试要点

**前置动作（每个前端文档生成前必须执行，顺序不可调换）**：

⛔ **第一步：收集补充接口（Step 2.5 新增）**
- 读取 `_PROGRESS.md` 中所有后端模块的「补充接口（Step 2.5）」列表
- 筛选与本前端页面域相关的补充接口，记录到本文档待纳入清单

⛔ **第二步：提取原始接口（架构文档已有 + 后端详设已生成）**
- **逐个打开**本前端页面域依赖的所有后端详设文档（从架构文档或 `_PROGRESS.md` 中识别关联模块）
- **逐个读取**每份后端详设文档的第3节「接口定义（OpenAPI 3.0 YAML）」全文
- **逐接口提取**以下信息到本文档待纳入清单：
  - HTTP 方法 + 完整路径（含路径参数占位符，如 `/api/users/{id}`）
  - `requestBody.content.application/json.schema.properties` 中的所有字段名（含嵌套字段，保留大小写）
  - `parameters` 中的 query/path 参数名称
  - `responses.200.content.application/json.schema.properties` 中的所有响应字段名（含嵌套字段，保留大小写）

⛔ **第三步：合并接口清单并去重**
- 将第一步的补充接口 + 第二步的原始接口合并为**完整接口清单**
- 同一路径 + 方法的接口保留最新版本（补充接口优先，因可能是对原始接口的修正）

⛔ **第四步：填写第3节 API 调用映射表格**
- **URL 列**：从完整接口清单中复制完整路径（禁止简写或省略路径参数）
- **请求参数/Body 列**：从第二步提取的 `requestBody.properties` 或 `parameters` 中复制字段名（**字段名大小写必须完全一致，禁止驼峰转换或自行发明**）
- **响应关键字段 列**：从第二步提取的 `responses.200.properties` 中复制字段名（**字段名大小写必须完全一致**）
- **禁止**使用通用占位符（`{field1, field2}`、`data.list`），必须使用后端实际定义的字段名

⛔ **第五步：字段级双向一致性核对**

> **核心原则：接口路径存在 ≠ 字段对齐。必须同时做正向和反向两个方向的核对。**

**5A. 正向核对（前端→后端）：前端写了的字段，后端必须有定义**
- 逐接口核对：前端详设第3节表格中填写的每个字段名，在对应后端详设第3节 OpenAPI YAML 中有对应定义
- 若发现不一致（如前端用 `userName` 但后端定义为 `user_name`），**停止生成，向用户报告不一致项**，等待用户确认采用哪个版本后再继续

**5B. 反向核对（后端→前端）：后端已定义的关键字段，前端是否全部纳入**
- 逐接口核对：对应后端详设第3节 `responses.200.schema.properties` 中的**每一个字段**，检查是否已在前端第3节「响应关键字段」列中出现
- **判定规则**：
  - 后端字段为 UI 展示类（`name`、`status`、`amount`、`createTime` 等）→ 前端未纳入则标记 `⚠️ 疑似遗漏字段`，**向用户确认是否需要展示**，确认后补入再继续
  - 后端字段为**枚举类型**（`orderStatus`、`payType` 等）→ **必须**在前端「响应关键字段」列中纳入（因枚举驱动前端逻辑分支），未纳入则 **停止生成，向用户报告**
  - 后端字段为纯内部字段（`tenantId`、`version`、`deletedAt` 等）→ 前端未使用可接受，无需报告
- **请求体反向核对**：后端 `requestBody.required[]` 中列出的必填字段，前端第3节「请求参数/Body」列中必须全部存在；缺失任一必填字段 → **停止生成，向用户报告**

**第3节 API 调用映射的 URL/方法/字段名必须来自后端详设第3节 OpenAPI 定义，禁止自行发明。**
输出路径：`doc/detailed/前端_{页面域}.md`

**微信小程序**（LC-MP-001 ≠ `无` 时执行）：
加载 `templates/frontend-template.md` Part 2 + `resources/design-guides.md` Part 4。
10个必填节：页面清单（含分包/tabBar归属）/ 页面交互规则 / API调用映射 / 状态与存储 / 微信能力使用 / 生命周期钩子 / 错误处理 / 分包策略 / 缓存策略 / 测试要点
**前置动作（与 Web 前端完全相同的五步流程，其中第五步包含 5A 正向核对 + 5B 反向核对）**：收集补充接口 → 提取原始接口（读取后端详设第3节 OpenAPI YAML）→ 合并去重 → 填写第3节 API 调用映射表格（字段名完全一致）→ 双向字段核对（5A 不一致停止报告；5B 枚举字段遗漏停止报告，展示字段遗漏询问用户，请求体必填字段缺失停止报告）。
输出路径：`doc/detailed/小程序_{页面域}.md`

⛔ **第六步：三端端间字段对齐核查（三端架构时必须执行）**

> **触发条件**：端类型为「三端（后端+Web前端+小程序）」时，在**小程序文档写入前**强制执行；「后端+小程序」双端架构时跳过。

**背景**：前端与小程序各自独立从后端复制字段，可能对同一接口形成不同版本的字段使用集合，导致两端字段不一致（如前端用 `nickName`，小程序用 `nick_name`，均自称"来自后端"但互不对齐）。

**执行步骤：**
1. 读取所有已生成的 `doc/detailed/前端_{页面域}.md`，提取第3节「API 调用映射」中每个接口的请求字段和响应字段清单
2. 从当前小程序文档第3节（第四步已填写）提取相同接口的字段清单
3. 逐接口比对相同后端接口在前端与小程序侧的字段差异：
   - **字段名不一致**（前端 `userName`，小程序 `user_name`）→ **停止生成，向用户报告差异项**，确认统一命名后再继续
   - **枚举字段覆盖的枚举值不同**（前端 switch 分支 ≠ 小程序 switch 分支）→ **停止生成，向用户报告**
   - **一端有某响应展示字段，另一端完全未纳入**→ 标记 `⚠️ 端间字段覆盖差异`，询问用户是否两端均应使用
4. 所有差异确认修正后，方可写入小程序文档

**失败动作（三端架构）**：若前端文档尚未全部写入，导致无法提取前端字段清单，**停止小程序文档生成**，告知用户须先完成全部前端文档再生成小程序文档。三端架构强制执行顺序：**后端全部完成 → 前端全部完成 → 小程序**。

> ⛔ **Step 4 同样遵守单文档节奏规则**：每次只生成一份前端/小程序文档，写入并更新 `_PROGRESS.md` 后必须暂停，输出进度确认消息（同 Step 3 格式），等待用户回复「继续」后方可生成下一份。

**失败动作**：若后端详设未生成完毕就进入 Step 4，立即停止，告知用户后端详设文档未完成，要求先完成 Step 3 再继续。
