# PART 1：访谈与提问框架

在澄清阶段加载本部分，用于指导结构化需求访谈。

---

## 1. 5W1H 框架（核心提问）

以此作为每轮首次访谈的骨架，从以下类别中选取 5~8 个针对性问题提问。

### Who（用户）
- "这个功能是为谁设计的？谁会用？谁会受益？"
- "是所有用户都能用，还是特定角色（如 VIP、管理员）？"
- "用户现在的替代方案是什么？他们在用什么笨办法解决？"

### What（内容 / 问题）
- "用户真正想解决的核心问题是什么？是表面需求还是深层需求？"
- "请描述一下这个功能成功运行后的样子？"
- "如果不做这个功能，最坏的结果是什么？"

### Why（目的）⭐ 最重要
- "为什么我们一定要现在做这个？它解决了哪个业务指标？"
- "这个需求背后的原始驱动力是什么？（老板拍脑袋？竞品有？用户调研？）"

### When（时机）
- "用户希望在什么时间点用到这个功能？（特定节日还是日常通用？）"
- "这个需求的时效性如何？短期促销还是长期基建？"

### Where（地点 / 场景）
- "用户在什么环境下使用？（嘈杂工厂还是安静办公室？WiFi 还是 5G？）"
- "是在 PC 端用，还是在手机端？手机端是站着用还是躺着用？"

### How（方式）
- "用户希望怎么完成这个操作？（三步完成还是一键完成？）"
- "目前业务的流程是怎样的？新流程会改变现有习惯吗？"

## 2. 边界与异常框架（反向提问）

使用以下问题挖掘用户常常忘记提及的边界情况和错误场景。

- **逆向思维**：「如果用户操作到一半退出了，数据怎么处理？」
- **极限思维**：「如果瞬间有 1 万人同时点击这个按钮，系统扛得住吗？」
- **无数据情况**：「如果接口返回失败，或者网络超时，页面给用户看什么？」
- **边界情况**：「如果用户输入了表情符号、1000 个字、负数，怎么办？」
- **权限边界**：「如果普通用户误操作了管理员才能做的事，系统怎么拦截？」

---

## 3. KANO 模型（优先级评估）

| 需求类型 | 判断问题 | 处理策略 |
|---------|---------|---------|
| **基本型**（底线） | "如果没有这个功能，你是不是会很不满意？" | 必须做，不做就失败 |
| **期望型**（投入产出比最高） | "有这个功能，你是不是会更满意？没有就不满意？" | 优先做，ROI 最高 |
| **兴奋型**（加分项） | "有这个功能会让你惊喜吗？没有也不会觉得有问题？" | 有余力再做 |

## 4. 需求冲突处理框架 ⭐

### 4.1 冲突类型识别

| 冲突类型 | 典型表现 | 示例 |
|---------|---------|------|
| **目标冲突** | 不同角色对成功的定义不同 | 产品要最大化用户增长，运营要控制成本 |
| **功能冲突** | 同一功能有互斥的实现方向 | A 说要强制实名，B 说要匿名保护隐私 |
| **优先级冲突** | 对功能重要性判断不一致 | 技术认为性能优化是 M，产品认为是 C |
| **资源冲突** | 需求超出时间/人力/预算约束 | 3 个月内要做 12 个月的功能量 |
| **技术可行性冲突** | 业务需求与技术实现存在矛盾 | 要求实时同步但网络延迟无法保证 |

### 4.2 冲突处理步骤

**Step 1：识别并命名冲突**
> 「我注意到这里存在一个冲突：[A方] 希望 [需求A]，而 [B方] 希望 [需求B]，两者在 [冲突点] 上互斥。」

**Step 2：记录冲突（在 PRD 草稿中标注）**
```
⚠️ 待决策项 [CONFLICT-001]
冲突描述：[简要描述冲突内容]
涉及方：[A方] vs [B方]
冲突点：[具体的互斥点]
备选方案：
  - 方案 A：[描述] → 优点：[...] 缺点：[...]
  - 方案 B：[描述] → 优点：[...] 缺点：[...]
决策人：[需要谁来拍板]
截止日期：[需要在什么时间前决策]
```

**Step 3：提供决策建议**
> 「从 [业务目标] 角度，我建议优先选择方案 A，因为 [理由]。但最终需要 [决策人] 确认。」

**Step 4：阻断生成，等待决策**
> 「在 [冲突点] 决策确认前，我将暂缓生成 [相关章节]，以避免后续返工。」

### 4.3 冲突记录模板（在 PRD 文档中使用）

```markdown
### 待决策项汇总

| 编号 | 冲突描述 | 涉及方 | 备选方案 | 决策人 | 状态 |
|------|---------|-------|---------|-------|------|
| CONFLICT-001 | [描述] | [A] vs [B] | 方案A / 方案B | [姓名] | 🔴 待决策 |
| CONFLICT-002 | [描述] | [A] vs [B] | 方案A / 方案B | [姓名] | 🟢 已决策：方案A |
```

## 6. 端类型专属访谈问题

### 6.1 Web 前端专属问题

- **页面范围**：「核心页面有哪些？请列举主要的页面或功能模块（如：首页、列表页、详情页、个人中心）。」
- **设计资产**：「是否有现有的设计稿、品牌色规范或竞品参考？还是需要从零设计？」
- **访问终端**：「用户主要在 PC 浏览器还是手机浏览器上使用？还是两者都需要支持？」
- **权限边界**：「是否有多种角色？不同角色看到的页面或可操作的功能是否不同？」
- **国际化**：「是否需要支持多语言？」

### 6.2 微信小程序专属问题

- **底部导航**：「底部导航栏有哪几个主要入口？（如：首页、分类、购物车、我的）」
- **登录方式**：「用户如何登录？是微信账号一键登录，还是手机号/账号密码登录？」
- **支付需求**：「是否需要在小程序内完成支付？主要支付场景是什么？」
- **消息通知**：「是否需要在特定事件发生后通知用户？（如：订单状态变更、活动提醒）」
- **分享需求**：「是否需要支持分享给好友或分享到朋友圈？分享的内容是什么？」
- **位置需求**：「是否需要获取用户地理位置？用于什么业务场景？」
- **离线需求**：「用户在弱网或无网络时，是否需要支持基本功能？」

### 6.3 多端并存专属问题

- **功能差异**：「各端的功能是完全一致，还是有差异化功能？（如：小程序只做下单，Web 后台做管理）」
- **用户群体**：「各端的目标用户是否不同？（如：小程序面向 C 端用户，Web 面向 B 端管理员）」
- **上线优先级**：「如果资源有限，哪个端优先上线？」
- **数据一致性**：「各端操作的数据是否需要实时同步？对数据一致性有什么业务要求？」

### 6.4 纯后端专属问题

- **调用方**：「这个后端服务的调用方是谁？（内部其他业务系统、外部合作伙伴、用户端应用、定时自动任务？）」
- **交互方式**：「外部系统或用户端需要主动查询数据，还是由本服务主动推送业务通知？数据交换是实时的还是批量的？」
- **数据量级**：「核心业务数据的规模预估是多少？（百万级、千万级？）业务方对查询等待时间有什么要求？」
- **一致性要求**：「哪些业务操作不允许出现数据不一致（如扣款、库存扣减）？哪些可以接受短暂延迟后同步？」
- **安全与权限**：「哪些业务功能仅限内部系统访问？哪些对外部开放？是否需要识别调用方身份？」

# PART 2：PRD 填写指南

> 本部分包含每个章节的填写规范、示例和行业基准值。
> **加载时机**：仅在首次生成 PRD 时加载，后续迭代修改无需重新加载。

---

## 1. 执行摘要

**一句话定义**格式：`[产品名] 是一个 [目标用户] 用于 [核心功能] 的 [产品类型]，解决了 [核心痛点]。`

示例：
> 「在线教育平台」是一个面向 K12 学生和教师的直播课程系统，解决了传统线下教学地域限制和课程回放缺失的问题。

---

## 2. 项目背景与目标

目标必须符合 SMART 原则（具体、可衡量、可实现、相关、有时限）：

❌ 错误：「提升用户体验」
✅ 正确：「上线后 3 个月内，用户课程完课率从 40% 提升至 65%」

### 成功指标示例

| 指标名称 | 当前值 | 目标值 | 衡量方式 | 时间框架 |
|---------|-------|-------|---------|---------|
| 日活跃用户数（DAU） | 0 | 5,000 | 埋点统计 | 上线后 6 个月 |
| 课程完课率 | 40% | 65% | 后台统计 | 上线后 3 个月 |
| 用户满意度（NPS） | - | ≥ 40 | 问卷调查 | 每季度 |

---

## 3. 用户角色与场景

### 用户画像表示例

| 用户类型 | 描述 | 技术水平 | 使用频率 | 核心需求 |
|---------|------|--------|--------|--------|
| 学生 | 18-25岁在校大学生，习惯移动端操作 | 中等 | 每天 1-2 次 | 随时随地看课、做笔记 |
| 教师 | 30-50岁，有线下授课经验，不熟悉直播工具 | 低 | 每周 3-5 次 | 简单易用的直播工具、学生出勤统计 |
| 管理员 | 平台运营人员，负责课程审核和数据分析 | 高 | 每天多次 | 高效的内容管理和数据报表 |

### 用户旅程示例
```
学生购课旅程：
首页浏览课程 → 查看课程详情 → 加入购物车 → 微信支付 → 进入我的课程 → 观看直播/回放 → 提交作业 → 获得证书
```

## 4. 功能需求

### MoSCoW 优先级说明

| 标记 | 含义 | 判断标准 |
|-----|------|---------|
| **M（Must Have）** | 必须有 | 没有这个功能，产品无法上线或核心业务无法运转 |
| **S（Should Have）** | 应该有 | 重要但不是生死线，可以在第一个版本后补充 |
| **C（Could Have）** | 可以有 | 锦上添花，有余力再做 |
| **W（Won't Have）** | 不做 | 明确排除，防止范围蔓延 |

### 验收标准（AC）写法规范

> ⚠️ AC 只描述**业务行为和业务结果**，禁止出现响应时间、接口名称、HTTP 状态码、字段名等技术描述。

```
Given [业务前置条件]
When  [用户执行的具体业务操作]
Then  [业务预期结果，描述用户看到什么或业务状态如何变化]
```

**示例（用户登录功能）：**

AC-1（正常登录）：
```
Given 用户已注册且账号状态正常
When  用户输入正确的手机号和密码，点击「登录」
Then  登录成功，页面跳转至首页，顶部显示用户头像和昵称
```

AC-2（密码错误）：
```
Given 用户已注册
When  用户输入正确手机号但错误密码，点击「登录」
Then  页面提示「密码错误，还可尝试 N 次」，N 从 4 递减；连续错误 5 次后账号锁定 30 分钟
```

## 6. 数据需求

> ⚠️ 埋点描述只写**业务事件名称**、**触发业务场景**和**收集目的**，禁止出现技术字段名。

| 业务事件名称 | 触发业务场景 | 收集目的 |

## 7. 非功能需求（行业基准值）

> 以下基准值供 AI 理解业务规模量级，**不得直接复制到 PRD 文档中**。PRD 必须用**业务语言**描述。

### 7.1 性能基准值

| 业务规模 | 用户体验目标 | 业务并发规模估算 |
|---------|-----------|--------------|
| 小规模（DAU < 1万） | 操作秒级响应，用户无明显等待感 | 同时在线约 DAU 的 5%~10% |
| 中等规模（DAU 1万~10万） | 核心操作秒级响应，高峰期略有延迟可接受 | 同时在线约 DAU 的 10%~20% |
| 大规模（DAU > 10万） | 全时段秒级响应，大促期间保持稳定 | 同时在线约 DAU 的 20%+ |

**PRD 填写示例（业务语言）：**
```
- 预计用户规模：上线初期日活约 5,000 人，高峰期同时在线约 500 人
- 用户体验要求：核心业务操作（查询、提交）用户感知秒级响应，无明显等待
- 可用性要求：核心业务功能全天候可用，计划维护需提前通知用户
```

### 7.2 安全与合规要求

> ⚠️ PRD 中的安全需求只描述**业务保护目标**，禁止出现 HTTPS、TLS、BCrypt、JWT 等技术方案。

| 业务安全目标 | 填写示例 |
|-----------|---------|
| 数据传输安全 | 「所有用户数据传输须经过加密保护」 |
| 密码安全存储 | 「用户密码须以不可逆方式加密存储，禁止明文保存」 |
| 敏感信息脱敏 | 「手机号、身份证号等敏感信息展示时中间位隐藏」 |
| 访问权限控制 | 「系统按角色控制功能访问权限，越权访问须被拦截」 |
| 隐私合规 | 「需符合《个人信息保护法》，提供隐私政策和授权说明」 |

### 7.3 兼容性要求

| 端类型 | 填写示例 |
|-------|---------|
| Web（PC 端） | 「支持主流桌面浏览器（Chrome、Edge、Firefox、Safari）近两年版本」 |
| Web（移动端） | 「支持 iOS 和 Android 主流手机浏览器近两年版本」 |
| 微信小程序 | 「兼容微信客户端近两年版本，覆盖 95% 以上活跃用户」 |

## 8. 假设与风险

### 假设条件示例

| 假设编号 | 假设内容 | 影响范围 |
|---------|---------|---------|
| A-001 | 用户具备基本的智能手机操作能力 | 界面复杂度设计 |
| A-002 | 第三方支付服务商在项目上线前完成商户资质申请 | 支付功能上线时间 |
| A-003 | 直播服务商已完成采购，服务能力满足峰值并发需求 | 直播功能稳定性 |

### 风险清单示例

| 风险编号 | 风险描述 | 发生概率 | 影响程度 | 应对策略 |
|---------|---------|---------|---------|---------|
| R-001 | 第三方支付服务商审核周期超预期，导致支付功能延期 | 中 | 高 | 提前 4 周启动申请；备选方案：先上线线下转账方式 |
| R-002 | 直播并发超出服务商承诺上限 | 低 | 高 | 设置单场直播人数上限；提前确认弹性扩容协议 |

## 9. 约束与依赖

**业务约束示例：**
- 必须使用公司现有的统一支付平台
- 用户数据必须存储在境内服务器（数据主权要求）
- 不允许引入新的付费第三方服务（预算约束）

**合规约束示例：**
- 需符合《网络安全法》和《个人信息保护法》
- 直播内容需接入内容审核（涉及未成年人保护）

---

## 10. 发布策略

| 发布方式 | 适用场景 | 风险等级 |
|---------|---------|---------|
| 全量发布 | 小功能修复、文案调整 | 低 |
| 灰度发布 | 核心功能上线、重大业务改版 | 中 |
| A/B 测试 | 业务流程优化、转化率提升 | 低 |
| 功能开关 | 需要随时开关的业务功能 | 低 |

**灰度策略示例：**
```
灰度阶段一（第 1-3 天）：5% 用户，观察业务异常率 < 0.1%，超标立即回滚
灰度阶段二（第 4-7 天）：30% 用户，观察同上 + 用户投诉率 < 0.01%
全量发布（第 8 天）：工作日业务低峰期（如上午 10:00-11:00）
```

## 11. 版本号变更规则

格式：`vMAJOR.MINOR.PATCH`

| 段位 | 触发条件 | 示例 |
|------|---------|------|
| **MAJOR** | 需求根本性变化、文档推翻重写 | v1.0.0 → v2.0.0 |
| **MINOR** | 新增/删除章节，内容扩充 | v1.0.0 → v1.1.0 |
| **PATCH** | 局部修正、评审意见小改、错别字 | v1.0.0 → v1.0.1 |

**变更类型标记：** 🆕 新建 → v1.0.0 | 🐛 修正 → PATCH+1 | ✏️ 修改 → PATCH+1 | ➕ 新增 → MINOR+1 | 🗑️ 删除 → MINOR+1 | 🔄 重构 → MAJOR+1

**强绑定规则：** 头部"版本" = 变更记录最后一行版本；头部"最后更新" = 变更记录最后一行日期。
