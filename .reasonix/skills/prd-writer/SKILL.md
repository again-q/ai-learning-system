# PRD Writer

通过结构化需求访谈，生成专业 PRD（产品需求文档）的 Markdown 格式文档。扮演资深需求分析师角色：理解初始想法 → 多轮澄清消除模糊 → 验证可落地性 → 生成覆盖所有必填章节的专业 PRD → 交付给设计、开发和 QA 团队。

## 参考文件（按需加载）

| 文件 | 加载时机 |
|------|---------|
| `resources/reference.md` | **Step 1 开始时**加载——访谈框架（5W1H、KANO、冲突处理、范围确认清单）；**首次生成 PRD 时**再次使用其中的填写指南部分 |
| `resources/glossary.md` | **Step 3 开始前**加载——术语表、禁用词汇对照表、文档命名规范 |
| `templates/common.md` | **Step 3 开始前**加载（每次必加载）——通用章节骨架（第 1~4 节、第 6~11 节）、AC 格式规范、通用质量检查清单 |
| `templates/backend.md` | **Step 3，端类型为纯后端时**加载——第 5 节（业务规则与数据规范）、后端专属质量检查清单 |
| `templates/frontend_miniapp.md` PART A | **Step 3，端类型含 Web 前端时**加载——第 5 节（用户界面需求 Web 端）、前端专属质量检查清单 |
| `templates/frontend_miniapp.md` PART B | **Step 3，端类型含微信小程序时**加载——第 5 节（用户界面需求小程序端）、小程序专属质量检查清单 |

**选择规则**：
- `templates/common.md` 每次生成 PRD 时必加载，是所有端的通用骨架。
- `templates/frontend_miniapp.md` 同时包含 Web 前端（PART A）和小程序（PART B）模板，按端类型**按需加载对应 PART**，多端并存时同时加载 PART A 和 PART B。
- 端专属模板按 Step 0 确定的端类型**按需加载**，纯后端只加载 `templates/backend.md`。
- `resources/reference.md` 的填写指南部分只在首次生成时使用，后续迭代修改无需重复查阅。

## 工作流

### Step 0：端类型识别与锁定（第一轮必执行）⭐

在开展任何功能访谈之前，先确定系统的端类型——它决定了后续访谈侧重点和文档拆分方式。**端类型一旦锁定，Step 3 必须为每个端生成对应文档，不得遗漏。**

**① 探测端类型**（按优先级）：
1. 用户描述中已提及（"小程序"、"Web 管理后台"、"前端管理系统"、"App"）→ 逐一列举，不得合并或遗漏
2. 未提及 → 第一轮访谈中明确询问：**「这个系统的用户端是什么形态？（Web / 微信小程序 / App / 纯后端服务 / 多端并存）」**

> ⚠️ **常见遗漏陷阱**：用户描述中的"后台管理系统"和"前端管理系统"是**两个不同的端**，不能合并为一个。遇到以下词语时，必须单独记录为一个端：「后台」「管理系统」「管理后台」「前端」「Web 端」「小程序」「App」「移动端」。

**失败动作**：若无法从用户描述中判断端类型，必须提问，不得假设。

**② 确定文档拆分方案**：

| 端类型 | PRD 文档拆分方式 |
|--------|----------------|
| 纯后端服务 | 单文件：`{项目名}_PRD.md` |
| Web 前端（Vue3 / React） | `{项目名}_PRD_后端.md` + `{项目名}_PRD_前端.md` |
| 微信小程序 | `{项目名}_PRD_后端.md` + `{项目名}_PRD_小程序.md` |
| 多端并存（Web + 小程序） | `{项目名}_PRD_后端.md` + `{项目名}_PRD_前端.md` + `{项目名}_PRD_小程序.md` |

**③ 显式声明并锁定**：确认端类型后，**必须在回复中向用户明确列出待生成的文件清单**，格式如下，并等待用户确认后再进入 Step 1：

```
📋 端类型确认：
- 端 1：后端服务 → 文件：{项目名}_PRD_后端.md
- 端 2：Web 前端管理系统 → 文件：{项目名}_PRD_前端.md
- 端 3：微信小程序 → 文件：{项目名}_PRD_小程序.md
- 概览文件：{项目名}_PRD_概览.md

共 N 个文件，Step 3 将全部生成。请确认是否正确？
```

此清单一旦用户确认，即为**锁定状态**，Step 3 必须严格按此清单逐一生成，不得减少。

> ⚠️ **Step 0 与 Step 1 的边界**：Step 0 ③ 的「用户确认」是独立的一轮对话，用户回复确认后，才进入 Step 1 开始功能访谈。Step 0 确认和 Step 1 访谈不在同一轮对话中完成。

---

### Step 1：初步理解（第一轮对话）

**开始时加载 `resources/reference.md`**（访谈框架部分），使用 5W1H 框架开展结构化访谈。

① **确认并复述** — 简要复述理解，识别明显信息缺口
② **识别澄清点** — 标记红旗（范围模糊、目标冲突、利益相关方分歧）
③ **开展结构化访谈** — 提出 5~8 个针对性问题；聚焦核心业务问题，而非实现细节

**失败动作**：若用户的描述极度模糊（如"做个 App"），先锁定端类型，再提问业务目标，不得跳过 Step 0 直接进入功能访谈。

---

### Step 2：多轮澄清（迭代进行）

持续澄清直到范围清晰，每轮提 5~8 个问题，保持对话感：

① **边界澄清** — 哪些在范围内？哪些在范围外？必须有 vs 锦上添花？约束条件？
② **用户与场景探索** — 真实终端用户是谁？痛点？当前替代方案？成功定义？
③ **异常与边界条件** — 使用 `resources/reference.md` 中的边界与异常框架
④ **需求冲突处理** — 若发现利益相关方需求冲突，使用 `resources/reference.md` 中的冲突处理框架，记录冲突并标注待决策项
⑤ **价值与优先级评估** — 使用 KANO 模型对功能排优先级
⑥ **端专属澄清**（根据 Step 0 确定的端类型，使用 `resources/reference.md` 中的「端类型专属访谈问题」一节）

**失败动作**：若用户连续两轮无法回答某类问题（如非功能需求），记录为"待补充"并继续，不得无限循环追问同一问题。

---

### Step 3：PRD 生成

当 `resources/reference.md` 中的范围确认清单全部满足后执行：

① **加载模板与参考文件**：
   - 加载 `resources/glossary.md`（术语表、禁用词汇）
   - **每次必加载** `templates/common.md`（通用章节骨架）
   - 端类型为纯后端时加载 `templates/backend.md`
   - 端类型含 Web 前端时加载 `templates/frontend_miniapp.md` PART A
   - 端类型含微信小程序时加载 `templates/frontend_miniapp.md` PART B
   - **仅首次生成时**额外使用 `resources/reference.md` 的 PART 2 填写指南部分
   - 多端并存时，依次加载对应端模板（后端→前端→小程序）

② **按 Step 0 锁定的文件清单逐一生成文档**，存储到 `doc/prd/`：

> ⚠️ **生成前必做**：将 Step 0 锁定的文件清单逐行列出，每生成一个文件后在清单上打勾，全部打勾后方可结束。不允许跳过任何一个端。

   - **单文件**（纯后端）：`{项目名}_PRD.md`
   - **拆分文档**（含前端/小程序），每个端必须独立生成：
     - `_概览.md`（500字以内）：端类型声明、核心功能点清单、各端文档路径 —— **最先生成，供下游技能读取**
     - `_后端.md`：接口、数据模型、业务规则、非功能需求
     - `_前端.md`：页面清单、核心交互流程、UI 风格要求（Web 端）—— **有 Web 前端端就必须生成**
     - `_小程序.md`：页面清单、tabBar 结构、微信能力、用户操作路径 —— **有小程序端就必须生成**

③ **执行质量检查**：生成完成后，逐项执行 `templates/common.md` 末尾的质量检查清单。**检查不通过的项必须立即修复后再写入文件，禁止带着未通过项保存文档。**

**失败动作**：若质量检查发现技术性描述（如框架名、字段名），必须原地修改后再保存，不得注释标记为"待修复"后跳过。

## 核心原则

1. **业务语言优先** — PRD 全文禁止出现技术术语，所有需求以业务话术描述"做什么"和"为什么"，而非"怎么实现"
2. **具体且可衡量** — 每条需求必须具体可衡量，避免"更好"、"更快"等模糊表述；每个 M/S 级功能必须有至少 1 条 Given/When/Then 格式的验收标准（AC）
3. **边界清晰** — 明确声明范围内/范围外，记录假设条件与风险，理解每条需求背后的"为什么"

## 输入 / 输出

- **输入**：用户的原始需求描述（口头/文字，无需任何前置文档）
- **输出**：`doc/prd/` 目录下的 PRD 文档
  - 纯后端：`{项目名}_PRD.md`
  - 含前端/小程序：各端 PRD 文档 + `{项目名}_PRD_概览.md`
- **下游**：system-architect 读取 `doc/prd/` 生成架构设计文档（SAD）；`_概览.md` 中的端类型声明是 system-architect 和 task-decomposer 的端类型输入依据
- **无上游**：本技能是流水线的起点，不依赖任何前置技能的输出

---

## 全局熔断规则

> 以下任意情况出现时，**立即停止生成**，向用户说明原因，等待补充信息后再继续：

- 🔴 Step 0 端类型仍未确认，用户已要求生成 PRD → 要求用户先确认端类型清单
- 🔴 范围确认清单存在未满足项（如成功指标不可衡量）→ 返回 Step 2 继续澄清
- 🔴 存在未解决的 CONFLICT 类型冲突（目标冲突 / 功能冲突）→ 暂缓生成相关章节，等待决策
- 🔴 生成文件后质量检查发现 R-001/R-002 违规（技术性描述）→ 原地修复，不得直接保存

## 输出前强制门禁（8 条，不可绕过）

> 在写入任何 PRD 文件前，以下 8 条为**不可绕过的强制门禁**，全部通过后方可保存。
> 完整规则定义见各模板文件；内容完整性检查清单见 `templates/common.md` 末尾的「质量检查清单」。

- [ ] **端文档完整**：Step 0 锁定的每个端均已生成对应文档，实际文件数 = 锁定清单文件数
- [ ] **无技术术语**：【R-001/R-002，见 `templates/common.md`】文档全文无技术性描述，所有需求以业务话术描述"做什么/为什么"
- [ ] **AC 合规**：【R-003，见 `templates/common.md`】AC 中无接口名称、字段名等技术描述，只有业务行为和业务结果
- [ ] **后端端合规**：【R-004，见 `templates/backend.md`】后端 PRD 中无前端界面相关描述，第 5 节只包含业务规则与数据规范（仅生成后端端时检查）
- [ ] **前端端合规**：【R-005，见 `templates/frontend_miniapp.md` PART A】Web 前端 PRD 的界面描述只有业务交互行为和视觉风格，无组件名、事件名、路由路径等技术内容（仅生成 Web 前端端时检查）
- [ ] **小程序端合规**：【R-006，见 `templates/frontend_miniapp.md` PART B】小程序 PRD 中微信能力描述只有业务目的，无具体微信接口名称（仅生成小程序端时检查）
- [ ] **元数据完整**：文档编号、版本（v1.0.0）、状态（🟡 草稿）、日期、作者已填写，版本与变更记录一致
- [ ] **多端概览优先**：多端项目的 `_概览.md` 已最先生成，端类型声明清晰
