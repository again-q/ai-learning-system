# Part 1：技术选型参考

## 六大评估维度

在做出最终选择前，从以下维度对每个技术候选项进行评估：

| 维度 | 关键问题 |
|------|---------|
| **成熟度与稳定性** | 是否已生产就绪？是否持续维护？社区是否活跃？是否有大规模部署案例？ |
| **性能** | 能否满足所需吞吐量/并发？延迟是否可接受？在预期数据量下表现如何？ |
| **可扩展性** | 是否支持水平扩展？垂直扩展上限？是否支持数据分区/分片？是否对无状态友好？ |
| **运维复杂度** | 部署和运维是否简单？可观测性工具是否成熟？灾难恢复方案是否完善？团队学习曲线如何？ |
| **成本** | 许可证费用（开源 vs 商业）？基础设施占用？是否有隐性成本（出口流量、按席收费）？ |
| **团队技术储备** | 团队是否有实战经验？学习曲线是否合理？文档和社区支持是否充分？ |

## 决策原则

1. **开源优先** —— 优先选择开源方案，避免厂商锁定，降低许可证成本。
2. **务实优于完美** —— 使用经过验证的成熟技术，避免过度设计。
3. **可落地的设计** —— 理论上更优但团队无法运维的选择是糟糕的选择。
4. **可扩展性与性能** —— 为增长而设计，在瓶颈出现前提前规划。
5. **安全内建** —— 优先选择安全记录良好、补丁活跃的技术。

## 记录选型理由

对于每个选定的技术，在设计文档和 `tech-stack.json` 中记录：
- **是什么** —— 简要描述
- **为什么选它** —— 关键原因（参考上方维度）
- **为什么不选其他** —— 考虑过的 1~2 个备选方案及排除原因
- **已知权衡** —— 对已接受的弱点进行客观评估

## 技术目录

# Part 2：NFR 量化指标

> 在新建设计时加载，帮助将"高性能"、"高可用"等模糊描述转化为可测量的具体数字。
> 非功能需求必须量化，以便 task-decomposer 在详设中直接继承。

**量化指标模板**（根据实际业务填写，不要留空）：

| NFR 维度 | 量化指标示例 | 架构影响 |
|---------|------------|---------|
| **并发用户数** | 峰值 500 并发 | → 连接池大小、缓存策略、是否需要读写分离 |
| **响应时间** | P99 ≤ 500ms，P95 ≤ 200ms | → 是否需要 Redis 缓存、异步处理、CDN |
| **吞吐量** | 日处理订单 10 万笔 | → 是否需要消息队列、批处理、分库分表 |
| **可用性** | 99.9%（年停机 ≤ 8.7 小时） | → 是否需要主从复制、健康检查、熔断器 |
| **数据保留** | 日志保留 3 年，业务数据永久 | → 分区策略、归档方案、存储成本 |
| **安全合规** | 个人信息保护法 / GDPR | → PII 加密、审计日志、数据主权 |

> 如果 PRD 中没有明确的 NFR 指标，在澄清阶段主动询问，不要用"高性能"、"高可用"等模糊描述代替具体数字。

# Part 3：数据库设计规范

> 在生成或升级数据库设计章节时加载。语言特化的 ORM 框架选型、实体类规范详见 `resources/overlays.md`。

## RBAC：始终使用三表模型

当系统存在多个用户角色时，使用标准三表 RBAC 模型：

- `t_user` —— 仅存用户身份，不内嵌 `role` 字段
- `t_role` —— 角色定义（ADMIN、FARMER、LOGISTICS 等）
- `t_user_role` —— 多对多关联表

> **为什么**：在 `t_user` 中内嵌 `role` 字段会阻止动态角色分配，且无法为一个用户分配多个角色。三表模型是 RBAC 的最小完备实现。

## JSON 字段：谨慎使用

仅在以下情况使用 JSON 字段：
- 数据结构真正可变且不可预测
- 永远不需要对 JSON 内容进行过滤、排序或聚合
- 数据始终作为整体读写

**禁止将 JSON 字段用于**：需要单独查询的结构化记录列表、需要唯一性约束的字段、会出现在 WHERE 子句中的字段。

## 敏感字段：静态加密

对 PII 字段（手机号、身份证号、邮箱）：
1. 在主列存储**加密值**（AES-256-GCM）
2. 添加**哈希列**用于查找（加 UNIQUE 索引）
3. 哈希应加盐：`HASH(value + server_secret)`，防止彩虹表攻击

## 通用命名规范

| 规范项 | 规则 |
|--------|------|
| 表前缀 | `t_` |
| 软删除字段 | `deleted`（整型，0=正常，1=已删除） |
| 创建时间 | `created_at` |
| 更新时间 | `updated_at` |
| 操作人 | `created_by`、`updated_by` |
| 状态字段 | 整型，注释中列出所有枚举值 |

## 非功能性数据库设计要点

| 要点 | 说明 |
|------|------|
| **索引策略** | 为所有 WHERE/JOIN/ORDER BY 高频字段建索引；避免在低基数字段单独建索引 |
| **分区策略** | 日志类、时序类大表按时间范围分区；业务表超过 5000 万行时评估分表 |
| **连接池** | 明确最大连接数配置，与数据库服务器的 max_connections 对齐 |
| **慢查询** | 明确慢查询阈值（建议 ≥ 500ms 记录），并说明监控方案 |

# Part 4：安全设计检查清单

> 在生成或升级安全设计章节时加载。语言特化的框架实现方式详见 `resources/overlays.md`。

## 认证与 Token 策略（通用）

| 检查项 | 说明 |
|--------|------|
| **Token 机制** | 单 Token vs 双 Token（Access Token 短期 + Refresh Token 长期）。面向用户的应用优先选双 Token + 无感刷新。 |
| **Token 存储** | Token 在客户端存储在哪里？（localStorage、httpOnly Cookie、内存）各方案的安全权衡是什么？ |
| **Token 吊销** | 登出或安全事件时如何使 Token 失效？（黑名单、Token 轮换） |
| **会话管理** | 如何处理并发会话？是否限制同一账号的最大在线设备数？ |
| **具体实现** | → 参见 `resources/overlays.md` 对应语言的"认证与 Token 策略"节 |

## 敏感数据加密（通用）

| 检查项 | 说明 |
|--------|------|
| **密码** | 使用强哈希算法（bcrypt cost ≥ 12 或 Argon2）。禁止 MD5/SHA1。 |
| **PII 字段** | AES-256-GCM 静态加密。另存哈希列用于唯一性校验和查找。 |
| **第三方 OAuth Token** | 单向哈希，禁止存明文。 |
| **传输安全** | HTTPS/TLS 1.2+（优先 TLS 1.3）。 |
| **文件访问** | 对象存储使用短 TTL（≤ 1 小时）预签名 URL。 |
| **具体实现** | → 参见 `resources/overlays.md` 对应语言的"敏感数据加密"节 |

## API 安全（通用）

| 检查项 | 说明 |
|--------|------|
| **限流** | 按 IP 和用户分别限流。为敏感接口（登录、短信）指定阈值。 |
| **防重放** | 敏感写操作需 `X-Timestamp` + `X-Nonce` 请求头。时间戳偏差 > 5 分钟或 Nonce 已见过的请求直接拒绝。 |
| **输入校验** | 所有用户输入使用框架提供的校验机制。禁止 SQL 字符串拼接。 |
| **RBAC 执行** | 每个 API 端点必须声明所需角色。数据隔离规则必须明确。 |
| **具体实现** | → 参见 `resources/overlays.md` 对应语言的"API 安全"节 |

## 审计与合规（通用）

| 检查项 | 说明 |
|--------|------|
| **审计日志** | 所有写操作记录到审计日志表，含操作人、动作、资源、IP、请求体（脱敏）、结果和耗时。保留期 ≥ 3 年。 |
| **敏感字段脱敏** | 手机号、姓名、身份证号在日志和 API 响应中必须脱敏。 |
| **合规** | 注明适用法规（个人信息保护法、GDPR、食品安全法等）。 |

# Part 5：特殊集成设计规范

> 当系统集成特殊技术时，在架构文档模块设计章节之后添加专属章节（第 10 节）。

## 区块链集成（FISCO BCOS / Hyperledger Fabric）

专属章节必须包含：
- **上链策略**：链上 vs 链下存储决策及理由（推荐：数据存 MySQL，SHA-256 哈希上链）
- **智能合约代码**：完整 Solidity（或链码）源码，含注释
- **哈希计算规范**：参与哈希的字段、顺序、分隔符
- **异步上链流程**：完整时序图（业务操作 → MQ 消息 → 链消费者 → 上链 → 回写 tx hash）
- **节点规划**：节点数量、角色（共识/观察者）、机构分配
- **验证流程**：消费者如何验证数据完整性（重算哈希 → 调用 verifyHash() → 返回结果）

## 支付集成

专属章节必须包含：
- 支付服务商选型及理由
- 支付流程时序图（含回调/Webhook 处理）
- 幂等性策略（如何防止重复扣款）
- 退款流程
- 安全：签名验证、HTTPS 回调校验

## 文件存储集成（OSS / MinIO）

专属章节必须包含：
- Bucket 结构与命名规范
- 直传 vs 服务端转发决策
- 预签名 URL 生成与 TTL 策略
- 文件哈希计算（用于完整性验证和区块链存证）
- CDN 配置（如适用）

# Part 6：SAD 层协作边界

> 与 task-decomposer 协作时加载，明确两个技能的粒度分工，避免重复劳动。

| 章节 | system-architect（SAD 层）写到这里 | task-decomposer（详设层）从这里接手 |
|------|----------------------------------|----------------------------------|
| **数据库设计** | 核心表名 + 关键字段 + ER 关系描述 + 索引策略概述 | 完整 DDL（含所有字段、约束、索引）|
| **API 设计** | 端点列表 + HTTP 方法 + 功能描述 + 认证方案 | 完整 OpenAPI 3.0 Schema（含 Request/Response 字段定义）|
| **模块设计** | 模块划分 + 职责描述 + 模块间依赖关系 | 伪代码逻辑 + 算法细节 + 测试用例 |
| **前端路由/页面** | 页面清单 + 路由路径 + 权限要求（F2）；组件分层策略（F4） | 每个页面的交互规则 + API 调用映射 + 组件内部逻辑 |
| **小程序页面** | tabBar 结构 + 分包策略 + 微信能力集成方案 | 每个页面的生命周期钩子 + 缓存策略细节 + 错误处理代码 |

> **为什么这个边界很重要**：SAD 层写得太深（如完整 DDL、完整 OpenAPI Schema），task-decomposer 就是重复劳动；写得太浅，task-decomposer 就缺乏足够的输入。
