# Code Reviewer

对项目代码进行全面的代码质量评审，输出结构化评审报告，作为代码进入测试阶段的质量门禁。支持**纯后端模式**（五维度）和**全栈模式**（七维度），评审语言和框架规范由编码规范文档的 LC 语言契约决定。

## 参考文件（按需加载）

| 文件 | 加载时机 |
|------|---------|
| `resources/glossary.md` | **首次触发技能时**加载——核心术语、问题等级定义、评审维度术语、代码异味、安全漏洞类型、性能反模式，全流程保持术语一致性 |
| `resources/review-checklist.md` | **Step 1 评审执行开始前**加载——后端五维度通用检查清单（维度1通用命名原则 + 维度2~5完整检查项）+ **四种语言特化检查项**（Java/Python/Go/Node.js，按 LC-001 使用对应节） |
| `resources/frontend-review-checklist.md` | **全栈模式，Step 1 维度6开始前**加载——前端六维度检查清单（Vue3/React 双框架） |
| `templates/report-template.md` | **Step 2 生成报告前**加载——标准评审报告模板（含全栈扩展节、问题清单、变更记录） |

## 工作流

### Step 0：就绪检查、模式感知与范围确认

**① 全栈模式感知（优先执行）**

读取 `doc/detailed/编码规范.md`，检查 `LC-FE-001` 字段：

```
LC-FE-001 不存在                  → 纯后端模式（执行五维度评审）
LC-FE-001 存在且 frontend/ 存在   → 全栈模式（执行七维度评审）
LC-FE-001 存在但 frontend/ 不存在 → 警告：前端代码尚未生成，本次仅执行后端五维度评审
```

**② 扫描代码范围**

- **后端**：扫描 `src/` 目录，识别接口层、业务逻辑层、数据访问层、DTO、配置层、SQL 脚本（目录名依 LC-010 分层模型）
- **前端**（全栈模式）：扫描 `frontend/src/` 目录，识别 `types/`、`api/`、`stores/`、`views/`、`composables/`、`router/`（目录名依 LC-FE-010 分层模型）

**③ 加载参考文档**

按优先级读取评审基准：

1. `doc/detailed/编码规范.md`（**最高优先级**）— 提取 LC-001/LC-003/LC-004/LC-010/LC-FE-001/LC-FE-010 语言契约及 ER 约束条款
2. `doc/detailed/{模块}_编码指令.md` — 提取 CC 完整性契约、BR 业务规则
2. `doc/detailed/编码规范.md` — 项目编码规范
3. `doc/detailed/{模块}_{功能域}.md` — 对应详细设计文档（含第3节 OpenAPI 定义，全栈模式必须）

**文档缺失处理规则：**

| 缺失文档 | 影响维度 | 处理方式 |
|---------|---------|---------|
| `编码规范.md` | 维度1（编码规范） | 注明「使用通用规范（语言无关的通用命名原则和安全规范）」，**不得假设语言为 Java 或任何特定语言**，维度1 仅检查通用层（`review-checklist.md` § 维度1），不加载语言特化文件 |
| `编码规范.md` | 维度1（编码规范） | 注明「未找到项目编码规范，使用通用规范」，继续评审（通用规范已包含核心检查项） |
| `{模块}_{功能域}.md` | 维度2（业务逻辑一致性）、维度0（契约一致性，全栈模式） | **维度2 降级为「无法验证业务逻辑一致性」**（报告中标注 🔘 状态），仅检查代码内部一致性（如异常处理完整性、数据一致性），不对照业务规则；**全栈模式下维度0 降级为「无法验证契约一致性」**（报告中标注 🔘 状态），仅检查前后端各自的内部一致性 |

若 `编码规范.md` 不存在，注明「使用通用规范（语言无关的通用命名原则和安全规范）」并继续，**不得假设语言为 Java 或任何特定语言**。

**④ 确认评审范围**

向用户确认待评审文件列表、是否有设计文档、是否有特别关注的维度。**若满足以下全部条件，则视为「上下文已明确」，直接执行 Step 1，无需等待确认：① 用户在本轮对话中已明确说明「评审所有代码」或「评审 {模块名}」；② `src/` 目录已存在；③ `doc/detailed/编码规范.md` 已存在。不满足上述任一条件，必须向用户确认后再继续。**

### Step 0.5：增量评审识别

在进入 Step 1 前，检测是否为多轮评审场景：

**多轮评审触发条件**（满足任一）：
- `doc/review/` 目录下已存在本模块的评审报告（如 `{模块名}_代码评审报告.md`）
- 用户描述中包含"重新评审"、"修复后再看"、"第二轮"、"按报告改完了"等关键词

**多轮评审模式行为**：
1. 读取已有评审报告，提取上轮 P0/P1 问题清单
2. 优先核查上轮问题是否已修复（逐条对照代码）
3. 在 Step 2 报告中，新增"上轮问题修复情况"节，明确标注每条问题的修复状态（✅ 已修复 / ❌ 未修复 / ⚠️ 部分修复）
4. 输出路径加 `_R2`/`_R3` 后缀（如 `{模块名}_代码评审报告_R2.md`），**版本重置为 `v1.0.0`**（与 Step 2 输出路径规则一致）

**首次评审模式**：`doc/review/` 目录不存在本模块报告，按 Step 1 正常流程执行完整评审。

### Step 1：评审执行

按参考文件表声明的时机，加载 `review-checklist.md`（含通用层 + 语言特化节）；全栈模式额外加载 `frontend-review-checklist.md`；其他语言（非 java/python/go/nodejs）仅使用通用层，在报告中注明"暂无该语言特化检查项"。

维度1的检查项 = 通用层（review-checklist.md § 维度1）+ 语言特化层（review-checklist.md §【对应语言】特化检查项）

#### 【全栈模式专属】维度0：前后端契约一致性（最高优先级，必须最先执行）

以详细设计文档第3节 OpenAPI 定义为**唯一事实来源**，按以下四个方向检查：

**0.A 后端实现覆盖率（优先检查）**
- 从设计文档第3节提取**所有接口路径列表**（HTTP 方法 + 完整路径）
- 逐接口在后端代码（依 LC-010 分层：Controller/Router/Handler 层）中查找对应实现
- **若设计文档中存在某接口，但后端代码中无对应的路由注册/Handler 函数 → P0（接口未实现）**
- 输出未实现接口清单，格式：`❌ 未实现：{HTTP方法} {路径}（来源：{模块}_{功能域}.md § 第3节）`

**0.B 前端契约一致性**

> ⛔ 需同时检查类型声明层和实际使用层，不得仅检查 types/api 目录后即视为通过。

- `frontend/src/types/` — 逐字段对照 response schema（字段名大小写、类型兼容性、漏字段/幽灵字段）
- `frontend/src/api/` — 逐接口对照 paths（URL、HTTP 方法、requestBody 字段名、漏接口/幽灵接口）
- **绕过类型层检查**：扫描 `frontend/src/views/`、`frontend/src/stores/`、`frontend/src/composables/` 中直接引用响应字段的表达式（如 `res.data.xxx`、`response.xxx`），核对字段名是否与设计文档 response schema 一致
  - 发现字段名与 types/ 声明不一致的直接引用 → P0（字段名双重不一致）
  - 发现 views/stores 层使用了 types/ 未声明的字段名 → P1（绕过类型层直接使用）
- **枚举完整性检查**：凡 response schema 中类型为枚举的字段（`enum` 标注或字段名含 `Status`/`Type`/`State`），检查前端 `switch`/`if-else`/条件渲染是否覆盖设计文档定义的所有枚举值；缺少任一枚举 case → P1

**0.C 小程序契约一致性**（LC-MP-001 存在且 miniapp/ 目录存在时执行）

> ⛔ 检查粒度与 0.B 对齐：不仅检查接口路径，还须逐字段核对响应体字段覆盖和枚举完整性。

- `miniapp/utils/request.js` 或同等请求封装 — 逐接口对照 paths（URL、HTTP 方法、请求参数名）
- **响应字段逐字段核对**：扫描 `miniapp/` 各页面实际使用的响应字段名，逐字段对照设计文档 `responses.200.schema.properties`
  - 响应字段名大小写不一致（如小程序用 `user_name`，设计文档定义 `userName`）→ P0
  - 使用了设计文档未定义的字段（幽灵字段）→ P2
  - 未使用设计文档中 UI 展示类字段（疑似遗漏）→ P1
- **枚举完整性检查**：与 0.B 相同，检查小程序代码对枚举字段的条件分支是否覆盖全部枚举值；缺少任一 case → P1
- **必填字段发送检查**：对照设计文档 `requestBody.required[]`，检查小程序实际请求是否包含所有必填字段；缺少必填字段 → P0

**0.D 前端 vs 小程序端间字段一致性**（三端架构时必须执行：frontend/ 和 miniapp/ 同时存在）

> **背景**：0.B 和 0.C 各自与设计文档对照可能均通过，但两端对同一接口的字段使用方式不一致，导致运行时行为差异（如前端处理了退款状态、小程序未处理）。

- 基于 0.B 和 0.C 已建立的字段清单，逐接口进行前端 vs 小程序端间比对：
  - **同一响应字段，两端使用了不同命名**（如前端 `nickName`，小程序 `nick_name`）→ P0（端间字段命名裂变）
  - **枚举字段，前端覆盖的枚举值集合 ≠ 小程序覆盖的枚举值集合**（如前端处理了 `REFUND` 状态，小程序没有）→ P1（端间枚举处理不对称）
  - **前端使用了某 UI 展示字段，小程序完全未使用（或反之）** → P1，报告中标注「需人工确认：是否属于有意差异化设计？」

契约问题直接定级：
- **P0**：字段名/URL 不一致；后端接口未实现（0.A）；小程序缺少必填字段（0.C）；端间字段命名裂变（0.D）
- **P1**：类型不兼容；漏接口；枚举 case 不完整（0.B/0.C）；绕过类型层（0.B）；端间枚举不对称或展示字段覆盖差异（0.D）
- **P2**：幽灵字段/幽灵接口（代码中存在但设计文档未定义）

#### 维度1：编码规范符合性

依 LC-001 语言特化，检查命名规范、注解/装饰器完整性、工具库使用（依 LC-008）、日志规范、响应格式。
- **通用检查项**：见 `resources/review-checklist.md` § 维度1（命名原则、日志、响应格式、代码格式）
- **语言特化检查项**：见 `resources/review-checklist.md` §【Java/Python/Go/Node.js】特化检查项对应节

#### 维度2：业务逻辑一致性

对照详细设计文档，检查业务规则完整性（第2节）、伪代码忠实度（第4节）、接口契约符合性、异常处理完整性。详细检查项见 `resources/review-checklist.md` § 维度2。

#### 维度3：安全漏洞扫描

覆盖 OWASP Top 10 + 企业级常见漏洞：SQL/查询注入、越权访问、敏感信息泄露、认证与授权、输入验证、幂等性、并发安全与容错机制。详细检查项见 `resources/review-checklist.md` § 维度3。

#### 维度4：性能反模式识别

检查 N+1 查询、大事务、缓存问题（穿透/击穿/雪崩）、数据库查询效率（索引/SELECT*/深分页）、内存与资源管理。详细检查项见 `resources/review-checklist.md` § 维度4。

#### 维度5：可维护性评估

检查代码复杂度（方法行数/嵌套层级/类大小）、重复代码、魔法数字与硬编码、可测试性、注释质量。详细检查项见 `resources/review-checklist.md` § 维度5。

#### 【全栈模式专属】维度6：前端专项审查

依 LC-FE-001 框架（Vue3 / React），执行六个子维度：6.1 编码规范、6.2 业务逻辑一致性、6.3 安全检查（Token 存储/XSS/路由守卫）、6.4 性能反模式（懒加载/虚拟滚动）、6.5 可维护性、6.6 框架特化检查。详细检查项见 `resources/frontend-review-checklist.md`。

### Step 1.5：问题分级与汇总

| 等级 | 定义 | 是否阻断 |
|-----|------|---------|
| **P0 致命** | 安全漏洞、数据丢失、核心逻辑错误、占位实现（CC-001）、**前后端契约断裂** | **是，必须修复** |
| **P1 高风险** | 性能严重问题、业务规则遗漏、违反 ER-FORBIDDEN/BR-GLOBAL、路由无权限守卫 | 建议阻断 |
| **P2 改进建议** | 可维护性问题、轻微规范违反 | 否 |

**编码规范文档驱动的优先判定（高于通用规则）：**

| 违反项 | 等级 |
|--------|------|
| 违反 `ER-SECURITY` | P0 |
| 存在占位实现（`CC-001`）/ 伪代码步骤未翻译（`CC-002`） | P0 |
| 前后端字段名/URL 不一致 | P0 |
| Token 存储在 `localStorage` | P0 |
| **设计文档第3节定义的接口，后端代码中无对应实现（接口漏实现）** | **P0** |
| **小程序未发送设计文档 `requestBody.required[]` 中的必填字段** | **P0** |
| **前端与小程序对同一接口的响应字段使用了不同命名（端间字段命名裂变）** | **P0** |
| 违反 `ER-FORBIDDEN` / `BR-GLOBAL` / `ER-REQUIRED` | P1 |
| 路由无权限守卫（需登录页面） | P1 |
| 页面组件直接调用 `api/` 层（违反 LC-FE-010） | P1 |
| **枚举字段 case 不完整（前端或小程序少处理至少一个枚举分支）** | **P1** |
| **前端与小程序对同一接口的枚举字段处理不对称（端间枚举处理不对称）** | **P1** |
| 超出 `ER-QUALITY` 质量门槛 | P2 |

**通用安全与性能问题判定（无编码规范文档时生效）：**

| 问题类型 | 等级 | 典型案例 |
|---------|------|---------|
| SQL 注入漏洞 | P0 | `${}` 拼接用户输入、`WHERE id = " + id`、ORDER BY 字段无白名单 |
| XSS 漏洞 | P0 | `v-html`/`dangerouslySetInnerHTML` 无消毒、富文本无白名单过滤 |
| 越权访问 | P0 | 查询/修改/删除操作无数据归属验证（缺少 `WHERE user_id = ?`） |
| 敏感信息泄露 | P0 | 日志/响应/异常中含密码/Token/身份证/银行卡号 |
| N+1 查询 | P1 | 循环内调用 Mapper/Repository/ORM 查询 |
| 大事务 | P1 | 事务内含 HTTP 请求、MQ 发送、`sleep` 等待 |
| 缓存穿透/击穿 | P1 | 无缓存空值保护、热点 key 无分布式锁 |
| 无索引查询 | P1 | 查询条件字段无索引、`SELECT *`、深分页 |
| 幂等性缺失 | P1 | 支付/下单等关键写操作无幂等 key |

**评审结论标准：**

| 模式 | 结论 | 条件 |
|------|------|------|
| 纯后端 | ✅ 通过 | 无 P0，P1 ≤ 2 |
| 纯后端 | ⚠️ 有条件通过 | 无 P0，P1 > 2（**本次评审结束后 3 个工作日内**修复 P1 后可进入测试） |
| 纯后端 | ❌ 不通过 | 存在任意 P0 |
| 全栈 | ✅ 通过 | 后端 ✅ + 前端 ✅ + 契约一致 ✅ |
| 全栈 | ⚠️ 有条件通过 | 无 P0，但后端或前端存在多个 P1（**本次评审结束后 3 个工作日内**修复后可进入测试） |
| 全栈 | ❌ 不通过 | 任一栈存在 P0，**或契约断裂（P0）** |

> 注意：即使后端和前端各自通过，契约存在 P0 断裂时综合结论仍为 ❌ 不通过。

### Step 2：评审报告生成

评审报告是 gatekeeper 和 tester 的必要输入，**每次评审必须生成报告文件**，不适用条件触发策略。加载 `templates/report-template.md`，按模板生成结构化评审报告。

**输出路径：** `doc/review/{模块名}_代码评审报告{_RN}.md`（多轮评审加 `_R2`/`_R3` 后缀，版本重置为 `v1.0.0`）

**保存前强制检查：**
- [ ] 头部含文档编号（`CR-YYYYMMDD-NNN`）、版本（首次评审为 `v1.0.0`，多轮评审新建文件时重置为 `v1.0.0`，同一报告内修订按版本号变更规则递增）、状态（`🟡 草稿`）、变更记录
- [ ] 全栈模式：报告含"维度0：前后端契约一致性"节和"维度6：前端专项审查"节
- [ ] 多轮评审模式：报告含"上轮问题修复情况"节

## 关键原则

1. **契约优先于实现** — 维度0必须最先执行，前后端契约断裂是最高优先级问题。
2. **代码是唯一事实来源** — 评审基于实际代码，代码里没有的逻辑就是没有实现。
3. **安全问题零容忍** — SQL 注入、越权访问、Token 存 localStorage 等一律 P0，不因"概率低"降级。
4. **对照设计文档评审** — 有设计文档时，业务逻辑必须逐条对照，不凭经验判断"看起来对"。
5. **给出可操作的修复建议** — 每个 P0/P1 问题必须附带具体修复方案和代码示例。
6. **区分"错误"和"风格"** — P0/P1 是客观错误，P2 是建议，不把个人偏好包装成高优先级。

## 交互提示

1. **先确认范围** — 列出找到的代码文件和参考文档，请用户确认后再开始评审
2. **全栈模式先报契约** — 发现契约 P0 问题时立即提示，不等报告生成才告知
3. **分模块推进（强制暂停）** — 项目含 **2 个及以上后端模块**时，逐模块评审，每完成一个模块的五维度评审后，必须输出如下格式的进度消息并**等待用户回复"继续"后**再评审下一个模块（防止上下文溢出导致后续模块评审被截断或跳过）：
   ```
   ✅ [{模块名}] 评审完成。发现：P0×{N}，P1×{N}，P2×{N}。
   📋 待评审模块：{列出剩余模块名}
   👉 请回复「继续」评审下一个模块，或「出报告」直接生成已完成模块的评审报告。
   ```
   **唯一例外**：用户在启动时明确说「全部模块一次性评审」，则视为批量授权，但发现 P0 时仍需立即暂停并告知用户。
4. **标注不确定项** — 无法从代码推断意图的地方，标注"需人工确认"
5. **优先报告 P0** — 发现 P0 问题时立即在评审过程中提示

## 输入 / 输出

- **输入**：
  - 后端代码：`src/`（gatekeeper 产出）
  - 前端代码：`frontend/`（全栈模式，gatekeeper 产出）
  - 小程序代码：`miniapp/`（三端架构， LC-MP-001 存在时）

- **通过时**：tester 读取 `doc/review/` 评审报告 + `doc/detailed/` 详设文档 + `src/` 代码，生成测试用例并执行测试
